The engine behind Power BI - DirectLake, storage modes, and XMLA.

Concepts - read these first

⚡
DirectLake = no import, no DirectQuery lag

DirectLake reads Delta files directly from OneLake. Data stays where it is but Power BI reads it as fast as an import model. It falls ← to DirectQuery when a Delta table is not yet framed. The exam tests when the fall← happens.

📦
Large storage format unlocks XMLA write operations

To use the XMLA endpoint for write operations like scripted deployments, the workspace needs Large semantic model storage format enabled. Small format gives read-only XMLA. This is a common trap - people assume Premium capacity alone is enough.

🗃️
Star schema: fact tables hold numbers, dimensions hold context

A fact table stores measurements like sales amount. Dimension tables store context like dates and customers. They connect on surrogate 🔑s. Always set surrogate 🔑 columns Summarize By to None - you never want Power BI summing an ID column by accident.

🔀
cache.abf holds cached data - keep it out of Git

When you save a .pbip project, the semantic model 🗃️ contains cache.abf which holds actual data values. Add it to .gitignore. The model.bim file holds the schema definition - that is fine to commit.

🔑
Add managed identity to the app, not the workspace

When an Azure app needs to access the XMLA endpoint using its own identity, you add a managed identity to the app and grant that identity workspace access. Adding it to the workspace alone does not give the app its own credential.
