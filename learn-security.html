🔐 Security & Governance

Workspace roles, RLS, sensitivity labels, domains, and Purview.

Core concepts first

🏢
Workspace roles: Admin > Member > Contributor > Viewer

Admin: full control, can add/remove users, delete workspace. Member: can publish and share. Contributor: can create and edit items (like a Lakehouse or shortcut) but can't share. Viewer: read-only. The exam often tests the minimum role needed for a specific task — always go for the least-privilege answer.

🔒
RLS = filter rows based on who's asking

Row-Level Security limits what rows a user sees in a semantic model. You define a role with a DAX filter expression. Dynamic RLS uses USERNAME() or USERPRINCIPALNAME() to filter based on the logged-in user's email — so one role handles all users automatically.

🏛️
Domains group workspaces — only Fabric Admins can create them

Fabric Domains let you logically group workspaces (e.g. by department). Only a Fabric Admin can create a domain. A Domain Admin (assigned by the Fabric Admin) can then manage that domain — but cannot create new ones. This hierarchy trips up a lot of exam candidates.

🏷️
Sensitivity labels travel with the data

Microsoft Purview sensitivity labels (Confidential, Highly Confidential, etc.) apply to Fabric items and carry over when data is exported — to Excel, PDF, or elsewhere. Endorsement (Promoted/Certified) is different — it's about data quality and trust, not confidentiality. Know the difference.
