Make your pipeline smart — dynamic expressions

Same pipeline. Different behavior. No rewriting. That's the power of parameters and expressions.

~5 min read · 1 exam question · 3 quiz questions

01 — The story

What's the actual problem?

Many sources, changing conditions, growing complexity. You need a pipeline that adapts.

You have a Fabric workspace with data coming from many different sources. New sources get added. File names change. Conditions shift. Your pipeline needs to handle all of this without being rewritten every time.

The analogy that sticks

Think of it as a same robot, different instructions. The robot (pipeline) doesn't change. But you give it a note (parameter) that says "today, go to this source, use this file name, write to this table." Tomorrow, different note — same robot.

02 — How it works

Brain first, then action.

Dynamic expressions sit between the pipeline and the activity. They decide HOW things run.

03 — What it actually looks like

Real examples. Super simple.

Expressions use @{} syntax. They resolve at runtime.

Dynamic file path @concat('sales_', formatDateTime(utcNow(), 'yyyy-MM-dd'), '.csv')

→ Produces: sales_2026-04-08.csv
Dynamic source table (using parameter) @pipeline().parameters.sourceTable

→ Could be: orders, customers, products — depends on what's passed in
Conditional logic @if(equals(@pipeline().parameters.region, 'EU'), 'eu-lakehouse', 'us-lakehouse')

→ Routes data to different lakehouses based on region

04 — The shift

Without vs with dynamic expressions

Without = rigid. With = flexible. That's the whole exam point.

Without (rigid)

Hardcoded everything

One pipeline per source. New source = new pipeline. File name changes = rewrite. Doesn't scale.

With (flexible)

One pipeline, many behaviors

Parameters change at runtime. Same pipeline handles SQL, CSV, APIs. New source = new parameter value. No rewriting.

05 — Exam traps

Three traps. Know why each fails.

Every wrong answer is a rigid solution to a flexibility problem.

Trap: "Standardize everything"

Standardize all sources to SQL Server

Simplifies config, but kills flexibility. Forces every source into one format. Can't handle diverse data types. The opposite of what they're asking for.

Trap: "Use one tool for everything"

Copy Data activity exclusively

Copy Activity is great for moving data, but it only moves — it doesn't adapt logic. Can't do preprocessing, conditional routing, or transformation. Using it exclusively limits the pipeline.

Trap: "Code-first approach"

Azure Functions for all sync

Functions handle small jobs and events well. But not suited for comprehensive data integration at scale. Can't orchestrate full pipelines or handle robust transformations for real-time analytics.

The common mistake

You might pick "standardize" because it sounds clean and organized. But the question is asking for flexibility, not uniformity. When things vary → use dynamic logic.
