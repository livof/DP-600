Streaming data, Eventstream, KQL queries, and real-time dashboards.

Concepts - read these first

🌊
Eventstream then KQL Database then Real-Time Dashboard

This is the real-time stack in Fabric. Eventstream ingests live events. KQL Database (inside an Event🏠) stores and queries them. Real-time dashboards visualise the stream. Each piece has a distinct role - know which does what.

🔍
KQL uses pipes to chain operations

KQL (Kusto Query Language) chains steps with a pipe character: TableName | where TimeStamp > ago(1h) | summarize count() by Category. It is optimised for time-series and log data. The exam tests basic KQL operations, not advanced scripting.

⏱️
ago() is KQL's relative time filter

ago(1h) means 1 hour ago from now. ago(7d) means 7 days ago. Used constantly to filter recent data: | where ingestion_time() > ago(30m). The exam asks you to complete or interpret KQL snippets like this.

🏠
Event🏠 is the container. KQL Database is the data store.

An Event🏠 is a Fabric item that holds one or more KQL Databases. You create an Event🏠, then create KQL Databases inside it. Eventstream destinations point to a KQL Database, not directly to an Event🏠.
