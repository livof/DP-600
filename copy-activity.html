05 — Exam traps

Three traps. Know them cold.

The exam loves these. Here's exactly why each one fails.

Trap: "Looks similar"

Synapse Pipelines

Sounds right — it does batch processing and ingestion. But it's Azure Synapse Analytics, not Fabric. Extra orchestration overhead, more latency. The exam wants the Fabric-native tool.

Trap: "Fast tool"

BCP (Bulk Copy Program)

Yes it's fast. But it's a command-line utility — manual, exports to files first, no real-time support, can't handle complex transformations. No orchestration. Dead end for lakehouse integration.

Trap: "Integration tool"

Azure Logic Apps

Great for event-driven automation — emails, triggers, webhooks. But not built for high-throughput data movement. Lacks the performance features of Copy Activity in Fabric pipelines.

06 — The deeper concept

ETL vs ELT

Fabric = ELT. Load first, transform later. Copy Activity is the loader.

ETL (old)ELT (Fabric)
OrderTransform → LoadLoad → Transform
SpeedSlower — waits for transformFaster — raw data lands immediately
FlexibilitySchema locked earlySchema-on-read, reshape anytime
Fabric tool—Copy Activity (load) → Spark/SQL (transform)
Copy Activity = the "L" in ELT

It loads raw data fast using binary copy. You transform later with Spark, Dataflows, or SQL. Load fast, think later. That's the modern way.

08 — Real exam question

Practice with the actual question

This is from the DP-600 practice assessment. Tap to answer.

You have a Fabric tenant with a lakehouse designed to store data from various sources. You need to integrate sales data from an OLTP system into this lakehouse. The sales data is critical for timely decision-making, and the integration must be efficient and reliable, with minimal impact on the source system's performance.

What should you do?

Use Azure Logic Apps.
Use the bulk copy program (BCP).
Configure a data pipeline with a Copy Data activity.
Use Synapse pipelines.
Correct: Configure a data pipeline with a Copy Data activity.

Copy Data activity in Fabric Data Factory is designed for large-scale data migrations. The binary copy method skips serialization/deserialization overhead, ensuring high throughput and low integration times. Minimal impact on the source OLTP system.

Why not BCP? It's a command-line tool that exports to files first. Manual, no real-time support, can't integrate directly into a lakehouse without extra steps.

Why not Synapse pipelines? They work for batch processing, but the overhead of Synapse orchestration adds latency. Fabric wants Fabric-native tools.

Why not Logic Apps? Built for event-driven workflows (emails, triggers). Not designed for high-throughput, efficient data movement into a lakehouse.

Q1: What makes Copy Activity faster than other ingestion methods for large datasets?

A. Binary copy method — skips serialization/deserialization
B. It compresses data before transfer
C. It uses a dedicated SQL connection
D. It transforms data during transfer
Binary copy moves data in its raw binary format. No time spent converting to/from intermediate formats. That's why it's fast — high throughput, low overhead.

Q2: What's the main advantage of ELT over ETL for lakehouse ingestion?

A. Data is cleaned before entering the lakehouse
B. Raw data lands fast, you transform later
C. It removes the need for a pipeline
D. It auto-creates star schemas
ELT = Extract, Load, Transform. Copy Activity does the E and L. Get raw data in fast. Transform later with Spark or SQL when you're ready. More flexible, more modern.

Q3: When should you use an OneLake shortcut instead of Copy Activity?

A. When you need to transform during ingestion
B. When you need a daily schedule
C. When you want to physically move data
D. When you want to reference data without copying it
Shortcuts = reference in place, zero copy. Copy Activity = physically move data. Key exam distinction: shortcut references, pipeline copies.
